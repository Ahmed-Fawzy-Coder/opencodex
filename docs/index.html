Codex provider routing, documented plainly

Use any model inside Codex.

opencodex is a local Responses API proxy for Codex. It routes provider models into the Codex catalog, preserves reasoning controls, and lets you choose the right sub-agent surface for each new session.

Install and start npm i -g @bitkyc08/opencodex && ocx start

Session shape

Sub-agent Surface

The multi-agent surface is selected per new Codex session. Use ocx v2 mode v1, ocx v2 mode default, or ocx v2 mode v2 to choose how opencodex writes model metadata into the Codex catalog.

v1 classic

Classic single-thread agent mode. A session uses one model, runs tool calls sequentially, and keeps the familiar v1 execution surface even when upstream model pins would prefer v2.

base upstream defaults

The install default. GPT-5.6 Sol and Terra use v2, Luna uses v1, and every other model follows Codex's own multi_agent_v2 feature flag.

v2 concurrent

Multi-thread agent mode. Codex can use spawn_agent, parallel tool calls, and concurrent sessions, with thread count controlled by ocx v2 threads <n>.

Provider bridge

Model Routing

Models are exposed to Codex as namespaced ids such as anthropic/claude-opus-4-8 or xai/grok-4.5. If you omit the prefix, opencodex uses defaultProvider or auto-matches common model families.

Adapter translation

  • anthropic routes Anthropic Messages-style models and OAuth/key credentials.
  • cursor bridges Cursor model ids and effort-suffixed variants into Codex.
  • opencode-go and similar gateways fit through OpenAI-compatible routing when their endpoint speaks chat/completions semantics.
  • xai routes Grok models through the xAI provider with OAuth or key-based auth.

Catalog behavior

  • Live discovery fetches each provider's /models list and caches it briefly.
  • Configured models provide static allowlists or fallback entries when live discovery is disabled.
  • Sub-agent picks come from the first five featured routed or native models in subagentModels.
  • Disabled models stay hidden from Codex without deleting provider configuration.
{
  "defaultProvider": "anthropic",
  "providers": {
    "anthropic": {
      "adapter": "anthropic",
      "baseUrl": "https://api.anthropic.com",
      "authMode": "oauth",
      "defaultModel": "claude-sonnet-4-6"
    },
    "xai": {
      "adapter": "openai-chat",
      "baseUrl": "https://api.x.ai/v1",
      "authMode": "oauth",
      "defaultModel": "grok-4.5"
    }
  }
}

Thinking controls

Reasoning Effort

opencodex advertises the Codex effort ladder when a provider or model supports it, then translates that label to the provider's wire value. Unsupported requests clamp down to the closest supported tier instead of failing the session.

lowfast, light
mediumbalanced
highdeeper
xhighextra high
maxmaximum
ultradelegating

Wire clamp

ultra is a Codex client behavior: maximum reasoning plus proactive multi-agent delegation. Before a request reaches an old native or third-party provider, opencodex follows the upstream boundary and sends max on the wire.

Sub-agent spawns

max is always available for sub-agent spawns. Older saved ladders that stop at xhigh are healed when the provider map shows that the real upstream wire value already accepts max. 

{
  "providers": {
    "anthropic": {
      "reasoningEfforts": ["low", "medium", "high", "xhigh", "max"],
      "modelReasoningEffortMap": {
        "claude-opus-4-8": { "max": "max" }
      }
    }
  }
}

Catalog safety

Context Cap

The context cap lowers Codex-visible context windows for provider catalog entries. It keeps very large provider claims from changing Codex compaction behavior, while never raising a smaller live model context window.

Default

The built-in global cap value is 350000 tokens. A provider only gets capped when it is enabled in providerContextCaps; otherwise its live or configured catalog metadata is left alone.

Changing it

Set contextCapValue to change the global cap value, then enable caps per provider through providerContextCaps. For static per-provider or per-model metadata, use contextWindow or modelContextWindows on the provider itself. 

{
  "contextCapValue": 350000,
  "providerContextCaps": {
    "anthropic": 350000,
    "xai": 350000
  },
  "providers": {
    "cursor": {
      "contextWindow": 200000,
      "modelContextWindows": {
        "glm-5.2": 1000000
      }
    }
  }
}

First run

Getting Started

Install the CLI globally, start the local proxy, then use Codex normally. opencodex syncs routed models into Codex's catalog and restores native Codex on ocx stop. 

npm i -g @bitkyc08/opencodex
ocx start
Config lives at ~/.opencodex/config.json. After adding or editing providers, run ocx sync or restart the proxy so Codex sees the refreshed model catalog.
